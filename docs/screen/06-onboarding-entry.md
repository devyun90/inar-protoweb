## 3. 상태 분기

### 기본 (진입 선택)
- "홈 만들기" / "초대코드 입력" 2개 카드. 어느 쪽도 미선택 상태.

### 코드 입력 — 검증 전
- 시트 입력 필드 비어 있음. "코드 확인" 비활성(코드 형식 미충족).

### 코드 검증 — 성공 (PENDING + 이메일 일치)
- 1-C 확인 시트: 홈명, 초대자, 역할(보조양육자), invitee_email 표시. "합류하기" 활성.

### 코드 검증 — 이메일 불일치
- 입력 필드 에러: "초대받은 이메일(invitee@example.com)로 로그인해주세요"

### 코드 검증 — 만료/취소
- 에러: "만료되었거나 취소된 초대예요. 초대자에게 재요청해 주세요."

### 로딩
- 코드 확인 / 홈 생성 / 합류 API 호출 중 스피너.

### 에러 (네트워크/서버)
- 스낵바 "처리에 실패했어요. 다시 시도해 주세요." + 재활성.

## 4. 인터랙션

| 동작 | 대상 | 결과 |
|------|------|------|
| 탭 | 홈 만들기 카드 | 홈 생성 API → 아이 프로필 등록 화면 (생성자 ADMIN 자동 부여) |
| 탭 | 초대코드 카드 | 코드 입력 바텀시트 |
| 입력 | 코드 필드 | 형식 충족 시 "코드 확인" 활성 |
| 탭 | 코드 확인 | 초대 검증 API → 성공 시 확인 시트 / 실패 시 에러 |
| 탭 | 합류하기 | 초대 수락 API → 메인 (role_id=2 USER) |
| 뒤로가기 | 화면 | 약관/로그인으로 복귀 |

## 5. 권한별 차이

| 요소 | ADMIN (주양육자) | USER (보조양육자) |
|------|------------------|-------------------|
| 역할 결정 시점 | "홈 만들기" 선택 → `POST homes` 시 **role_id=1(ADMIN) 자동 부여** | "초대 수락" → `role_id=2(USER)` 부여 |
| 다음 화면 | 아이 프로필 등록 → **메인 직행** (온보딩 완료 직후 기기 등록 Alert 1회) | 메인 대시보드 직행 (아이 정보는 홈에서 상속) |

> 본 화면에서 사용자가 진입 방식을 고르는 순간 역할이 확정된다. 홈 생성자는 별도 권한 설정 없이 자동으로 주양육자(ADMIN)가 된다.

## 6. API 연동

### 6-1. 데이터 needs (구체적으로 — API 계약의 입력)

**읽기(Read):**
| 데이터 | 필드 | 타입 | 비고 |
|--------|------|------|------|
| 초대 검증 결과 | invite_code | string | 입력 코드 |
| | status | enum(PENDING/ACCEPTED/EXPIRED/CANCELLED) | 유효성 |
| | home_id | string | 대상 홈 |
| | home_name | string | 홈 표시명 |
| | inviter_name | string | 초대자 이름 |
| | invitee_email | string | 초대 지정 이메일 |
| | role_id | int | 부여될 역할(2=USER) |
| | expires_at | datetime | 만료 시각 |
| 로그인 사용자 | login_email | string | 이메일 일치 검증용 |

**쓰기(Write):**
| 액션 | 필드 | 타입 | 검증 | 비고 |
|------|------|------|------|------|
| 홈 생성 | home_name | string | 1~30자 | 생성자 role_id=1 자동 |
| 초대 검증 | invite_code | string | 형식 `INV-XXXXXXXX` | |
| 초대 수락 | invite_code | string | status=PENDING & 이메일 일치 | |

### 6-2. 엔드포인트 매핑 (TBD — backend spec에서 확정)

| 액션 | METHOD /path | 비고 |
|------|--------------|------|
| 홈 생성 | `TBD` (POST homes) | 응답에 home_id, 생성자 ADMIN 부여 |
| 초대 코드 검증 | `TBD` (GET invitations/{code}) | status·invitee_email 반환 |
| 초대 수락 | `TBD` (POST invitations/{code}/accept) | 이메일 불일치 시 403 |

## 7. 데이터·검증 규칙

| 입력 필드 | 필수 | 형식/범위 | 단위 | 에러 메시지 |
|-----------|------|-----------|------|-------------|
| home_name (홈 만들기) | Y | 1~30자, 공백만 불가 | — | "홈 이름을 입력해 주세요" |
| invite_code | Y | `INV-` + 8자 영숫자 | — | "초대코드 형식이 올바르지 않아요" |
| 이메일 일치 | Y | login_email == invitee_email | — | "초대받은 이메일({invitee_email})로 로그인해주세요" |
| 초대 상태 | Y | status == PENDING | — | "만료되었거나 취소된 초대예요. 초대자에게 재요청해 주세요." |

> 이메일 일치·상태 검증은 서버가 수행하며, 클라이언트는 반환 코드(403/410 등)에 따라 위 메시지를 표시. 불일치 메시지에는 서버가 내려준 `invitee_email`을 그대로 치환한다.

## 10. 접근성

| 항목 | 기준 |
|------|------|
| 터치 타깃 | 옵션 카드 높이 ≥ 88px, CTA·입력 ≥ 48px |
