# 공통 상태 패턴 — 에러 · 빈 상태 · 로딩 · 세션 만료

> 시나리오: user_scenario 시나리오 14 (에러 & 빈 상태), 17 (검색 결과 없음) · 범위: MVP
> 연계: [user_scenario.md](../user_scenario.md) · [screen_scenario.md](../screen_scenario.md) · [design rc.4](../1/design.md)
> 성격: **단일 화면이 아닌 앱 전역 재사용 패턴 정의서.** 모든 화면의 상태 분기는 본 문서를 SSOT로 참조한다.

| 항목 | 내용 |
|------|------|
| 위치 | 전역 (모든 탭·서브탭·바텀시트 공통 오버레이/인라인) |

## 1. 레이아웃 (상태별 ASCII)

## 3. 상태 분기

본 문서가 정의하는 4개 패턴은 모든 화면에서 다음 우선순위로 평가한다.

```
로딩(요청 중) → 에러(실패) → 빈 상태(성공+데이터0) → 기본(성공+데이터≥1)
                  └ 401이면 에러보다 우선하여 세션 만료 처리
```

### 기본 (데이터 있음)
- 각 화면 고유 콘텐츠 렌더 (본 문서 범위 밖)

### 빈 상태 (데이터 없음)
- 카테고리별 아이콘 + 표준 카피 + CTA (§9-1)
- CTA는 권한 가드 통과 시에만 노출 (예: 공지 추가는 ADMIN만)

### 로딩 (스켈레톤)
- 실제 콘텐츠 레이아웃과 동일한 골격으로 shimmer 표시 (레이아웃 시프트 방지)
- 최소 표시 시간 없음 (응답 즉시 전환), 단 300ms 이내 완료 시 깜빡임 방지용 지연 표시 권장

### 에러 (전체/부분 + 재시도)
- 전체 실패: `ErrorStateView` 전체 표시
- 부분 실패(특정 섹션만): `InlineErrorCard`로 해당 영역만, 나머지는 정상 유지
- 재시도 버튼 → 동일 요청 재호출

### 세션 만료 (401)
- 어느 화면에서든 401 수신 시 진행 중 작업 중단 → `SessionExpiredDialog` → 로그인 화면 (네비게이션 스택 초기화)

## 4. 인터랙션

| 동작 | 대상 | 결과 |
|------|------|------|
| 탭 | 재시도 버튼 | 실패한 요청 재호출, 로딩 스켈레톤으로 전환 |
| 탭 | 빈 상태 CTA | 해당 추가 플로우 진입 (기록/TODO 바텀시트, 검색 초기화 등) |
| 탭 | 필터 초기화 (검색 빈 상태) | 필터/키워드 리셋 후 재검색 |
| 탭 | "로그인 화면으로" | 토큰 폐기 + 스택 초기화 + `/login` 라우팅 |
| Pull-to-refresh | 에러/빈 상태 화면 | 요청 재시도 |
| 자동 | 401 인터셉트 | 사용자 입력 없이 세션 만료 다이얼로그 표시 |

## 5. 권한별 차이

| 요소 | ADMIN (주양육자) | USER (보조양육자) |
|------|------------------|-------------------|
| 에러/로딩/세션 만료 | 동일 | 동일 |
| 빈 상태 "첫 기록 추가" CTA | 노출 | 노출 |
| 빈 상태 "TODO 추가" CTA | 노출 | 노출 (작성 허용) |
| 빈 상태 "공지 작성" CTA | 노출 | **숨김** (열람만) |
| 빈 상태 "기기 등록" CTA | 노출 | **숨김** |
| 검색 결과 없음 "필터 초기화" | 노출 | 노출 |

> 권한으로 CTA가 숨겨지는 경우, 빈 상태는 아이콘+안내 문구만 표시하고 액션 버튼은 렌더하지 않는다.

## 6. API 연동

> **해당 없음** — 본 문서는 특정 엔드포인트를 호출하는 화면이 아니라, 모든 API 응답을 가로채 상태로 변환하는 **공통 에러 처리 규약**이다. 아래는 HTTP 상태코드 → 상태 패턴 매핑표.

### 6-1. HTTP 상태코드 ↔ 상태 패턴 매핑

| 분류 | 조건 | 상태 패턴 | 표시 컴포넌트 | 재시도 | 카피 참조 |
|------|------|-----------|---------------|--------|-----------|
| 네트워크 | 연결 실패/타임아웃 (응답 없음) | 네트워크 오류 | `ErrorStateView` (`wifi-slash`) | O | §9-1 ERR_NETWORK |
| 4xx | 400 Bad Request | 입력 오류 | 폼 인라인 메시지 (화면 고유) | X | 화면별 검증 메시지 |
| 4xx | 401 Unauthorized | 세션 만료 | `SessionExpiredDialog` (`clock`) | X (로그인 이동) | §9-1 SESSION_EXPIRED |
| 4xx | 403 Forbidden | 권한 없음 | 토스트/스낵바 (화면 고유) | X | "권한이 없는 작업이에요" |
| 4xx | 404 Not Found | 데이터 없음 | 빈 상태 또는 "삭제됨" 안내 | X | §9-1 EMPTY_* |
| 4xx | 408 / 429 | 요청 지연/과다 | 5xx와 동일 처리 | O | §9-1 ERR_SERVER |
| 5xx | 500/502/503/504 | 서버 오류 | `ErrorStateView` (`warning-circle`) | O | §9-1 ERR_SERVER |
| 성공+빈 | 200 + 빈 배열/객체 | 빈 상태 | `EmptyStateView` (카테고리 아이콘) | X | §9-1 EMPTY_* |
| 요청 중 | pending | 로딩 | `Skeleton*` | — | — |

### 6-2. 처리 원칙

- **401 우선**: 어떤 화면이든 401은 다른 에러 처리보다 우선 (전역 인터셉터).
- **에러 메시지 비노출**: 서버 원본 메시지·스택·쿼리는 사용자에게 노출하지 않고, §9-1 표준 카피로 치환 (steering 보안 규칙 준수).
- **부분 실패 격리**: 한 화면에 복수 요청이 있으면 실패한 영역만 `InlineErrorCard`로 격리, 성공 영역은 유지.
- **재시도 대상**: 네트워크/5xx/408/429만 "다시 시도" 제공. 401/403/404는 재시도 미제공.

## 8. 네비게이션

| 구분 | 경로 |
|------|------|
| 진입 | 임의 화면의 데이터 로딩 실패/빈 결과/세션 만료 시 인라인 전환 |
| 이탈 (재시도 성공) | 원래 화면의 기본 상태로 복귀 |
| 이탈 (빈 상태 CTA) | 기록/TODO 추가 플로우, 검색 초기화 등 |
| 이탈 (401) | 로그인 화면 (`/login`), 네비게이션 스택 초기화 |
| 딥링크 | 해당 없음 (오버레이/인라인 상태) |
| 뒤로가기 동작 | 에러/빈 상태: 이전 화면으로 일반 pop / 세션 만료 모달: 닫기 불가(로그인 이동만) |

## 9. 데이터 예시 (마이크로카피 표준안)

### 9-1. 각 상태별 표준 카피

식별자(`KEY`)는 영어, 표시 문구는 한글. 앱 전역에서 본 표를 i18n 키로 사용한다.

**에러 상태**

| KEY | 아이콘 | 제목 | 보조 문구 | 액션 |
|-----|--------|------|-----------|------|
| `ERR_NETWORK` | `wifi-slash` | 인터넷 연결이 끊겼어요 | 네트워크 상태를 확인하고 다시 시도해 주세요 | 다시 시도 |
| `ERR_SERVER` | `warning-circle` | 일시적인 오류가 발생했어요 | 잠시 후 다시 시도해 주세요. 문제가 계속되면 고객센터로 문의해 주세요 | 다시 시도 |
| `ERR_PARTIAL` | `warning-circle` | 이 정보를 불러오지 못했어요 | — (부분 카드, 1줄) | 다시 시도 |
| `ERR_FORBIDDEN` | — (스낵바) | 권한이 없는 작업이에요 | 주양육자에게 문의해 주세요 | — |

### 9-2. 카피 작성 가이드 (예시)

> 본 문서는 더미 데이터가 아닌 **카피 표준안**이 핵심이므로, §9-2는 작성 규칙 예시로 갈음한다.

```
[톤앤매너]
- "~어요/~해요" 체로 따뜻하고 부드럽게 (steering 톤앤무드: 따뜻한 모던)
- 사용자 자책 유발 금지: "오류가 발생했습니다" (X) → "일시적인 오류가 발생했어요" (O)
- 내부 정보 노출 금지: 에러코드/스택/쿼리 미표시 (보안 규칙)

[구조] 제목(무슨 일) → 보조(왜/어떻게) → 액션(다음 행동)
  예) "검색 결과가 없어요" → "검색어나 필터를 바꿔서 다시 찾아보세요" → [필터 초기화]

[길이] 제목 ≤ 16자, 보조 문구 ≤ 2줄
[행동 유도] CTA는 동사로 끝맺음: "기록 추가하기", "다시 시도"
```

## 10. 접근성

| 항목 | 기준 |
|------|------|
| 터치 타깃 | 재시도/CTA 버튼 최소 48x48px |
| 색 대비 | 제목 `#1c1b1b`, 보조 `#4f453f` on `#fcf9f7` ≥ 4.5:1 / 에러 아이콘 `#ba1a1a` 대비 확보 |
| 스크린리더 라벨 | 에러: "오류, {제목}. 다시 시도 버튼" / 빈 상태: "{제목}, {CTA} 버튼" / 세션 만료: "다시 로그인이 필요합니다" (role=alert) |
| 라이브 영역 | 에러·세션 만료는 `aria-live=assertive` 상당으로 즉시 안내, 로딩 스켈레톤은 "불러오는 중" semantic 처리 |
| 모션 | shimmer 애니메이션은 "동작 줄이기(reduce motion)" 설정 시 정적 회색 블록으로 대체 |
| 포커스 순서 | 아이콘(데코, 포커스 제외) → 제목 → 보조 문구 → 액션 버튼 / 세션 만료 모달은 포커스 트랩 |
